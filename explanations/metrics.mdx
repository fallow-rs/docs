## Complexity metrics

These metrics appear in [`fallow health`](/cli/health) output, both per-function (findings) and per-file (file scores).

### Cyclomatic complexity

**What it measures:** The number of linearly independent paths through a function's control flow graph. For structured programs (like JavaScript/TypeScript), this simplifies to 1 + the number of decision points (`if`/`else`, `switch` cases, `for`/`while`/`do` loops, ternary `?:`, logical `&&`/`||`, `catch` blocks).

| Range | Interpretation | Action |
|:------|:---------------|:-------|
| 1–10 | Simple, easy to test exhaustively | No action needed |
| 11–20 | Moderate — most functions should stay here | Review if growing |
| 21–50 | High, hard to test all paths | Split into smaller functions |
| 50+ | Very high — testing all paths is impractical | Refactor urgently |

**Default threshold:** 20

**Research:** Introduced by Thomas McCabe in ["A Complexity Measure" (1976)](https://doi.org/10.1109/TSE.1976.233837). McCabe's original recommendation was that functions above 10 require more rigorous testing. The practical upper bound of 20 is from [NIST Special Publication 500-235](https://www.nist.gov/publications/structured-testing-testing-methodology-using-cyclomatic-complexity-metric) (Watson & McCabe, 1996), which recommends limiting cyclomatic complexity to 10 for most modules, with an upper limit of 20 for exceptional cases.

**Example** — `_parse` in zod's `types.ts` has cyclomatic 59 and cognitive 113 across 308 lines. This is the kind of function that benefits from decomposition:

```ansi
● High complexity functions (22)
  src/types.ts
    :774 _parse
          59 cyclomatic  113 cognitive  308 lines
  src/locales/en.ts
    :4 errorMap
          51 cyclomatic   94 cognitive  145 lines
```

### Cognitive complexity

**What it measures:** How hard a function is to understand when reading top-to-bottom. Unlike cyclomatic complexity (which counts paths for *testing*), cognitive complexity measures *comprehension difficulty* — it penalizes structures that break linear reading flow: nesting depth, `break`/`continue` to labels, recursion, and sequences of logical operators.

These two functions illustrate the difference:

<CodeGroup>

```typescript High cyclomatic, low cognitive
// Cyclomatic: 6 (one per case)
// Cognitive: 1 (flat switch is easy to read)
function getStatusLabel(status: string): string {
  switch (status) {
    case "pending": return "Pending";
    case "active": return "Active";
    case "paused": return "Paused";
    case "cancelled": return "Cancelled";
    case "completed": return "Done";
    default: return "Unknown";
  }
}
```

```typescript Low cyclomatic, high cognitive
// Cyclomatic: 4 (one per if)
// Cognitive: 9 (nesting adds +2, +3 increments)
function processOrder(order: Order): string {
  if (order.items.length > 0) {            // +1
    if (order.customer.verified) {          // +2 (nesting)
      if (order.total > order.limit) {      // +3 (nesting)
        return "requires_approval";
      }
    } else {                                // +1
      return "unverified";
    }
  }
  return "empty";
}
```

</CodeGroup>

The `switch` has higher cyclomatic complexity (more paths to test) but is trivially readable. The nested `if` chain has lower cyclomatic complexity but is harder to follow — cognitive complexity captures this.

| Range | Interpretation | Action |
|:------|:---------------|:-------|
| 0–7 | Easy to understand at a glance | No action needed |
| 8–15 | Moderate — may benefit from extraction | Extract helpers if growing |
| 15+ | Hard to follow, multiple concerns likely | Refactor: extract, simplify, or decompose |

**Default threshold:** 15 (matches SonarSource's default rule)

**Research:** Developed by G. Ann Campbell at SonarSource. The full specification is available as a [white paper from SonarSource](https://www.sonarsource.com/resources/cognitive-complexity/). The threshold ranges above are fallow's guidelines based on SonarSource's default threshold of 15; the 0–7 "easy" range reflects common industry practice for functions that need no review.

### Complexity density

**What it measures:** Total cyclomatic complexity divided by lines of code. Normalizes complexity by file size so that a 500-line file with complexity 50 (density 0.1) and a 10-line file with complexity 10 (density 1.0) are compared fairly.

| Value | Interpretation | Action |
|:------|:---------------|:-------|
| < 0.3 | Low — typical for data files, configs, simple modules | No action needed |
| 0.3–1.0 | Moderate — normal application code | Review the densest functions |
| > 1.0 | Very dense — nearly every line is a decision point | Refactor: extract logic, simplify conditions |

This metric combines McCabe's cyclomatic complexity with the standard practice of normalizing metrics by size for fair cross-file comparison.

## File health scores

Per-file scores from [`fallow health --file-scores`](/cli/health). Zero-function files (barrel files, re-export files) are excluded to avoid degenerate scores.

### Maintainability Index (MI)

Fallow uses a **simplified Maintainability Index** adapted for JavaScript/TypeScript module graphs.

**Formula:**

```
fan_out_penalty = min(ln(fan_out + 1) × 4, 15)
MI = 100 - (complexity_density × 30) - (dead_code_ratio × 20) - fan_out_penalty
```

Clamped to \[0, 100\]. Higher is better.

| Range | Interpretation | Action |
|:------|:---------------|:-------|
| 70–100 | Good maintainability | Monitor |
| 40–70 | Moderate | Review periodically, address if declining |
| 0–40 | Poor | Prioritize for refactoring |

**Example** — `errorUtil.ts` scores 57.5 due to 100% dead exports and moderate complexity density. The dead code ratio (100%) is the biggest factor pulling the score down:

```ansi
● File health scores (5 worst)

   57.5    src/helpers/errorUtil.ts
           1 fan-in    0 fan-out  100% dead  0.75 density

   76.2    src/helpers/util.ts
          35 fan-in    0 fan-out   86% dead  0.22 density

   90.5    src/types.ts
          42 fan-in    3 fan-out    0% dead  0.20 density
```

<Accordion title="How fallow's MI relates to the classic Maintainability Index">

The original MI from [Oman and Hagemeister (1992)](https://doi.org/10.1109/ICSM.1992.242525) combines Halstead Volume, cyclomatic complexity, lines of code, and comment percentage. [Microsoft adapted it](https://learn.microsoft.com/en-us/visualstudio/code-quality/code-metrics-maintainability-index-range-and-meaning) to a 0–100 scale for Visual Studio.

Fallow's variant makes three changes for the JS/TS ecosystem:

1. **Replaces Halstead Volume with complexity density** — Halstead metrics require full type resolution, which fallow doesn't have (syntactic analysis only). Complexity density captures the same "how complex per unit of code" signal.
2. **Adds dead code ratio** — unused exports are a JS/TS-specific maintenance burden that the classic MI doesn't account for.
3. **Adds fan-out coupling penalty** — high import counts are a strong signal of maintenance difficulty in module-based codebases. The penalty uses logarithmic scaling (`ln(fan_out + 1) × 4`, capped at 15 points) — an engineering choice that reflects diminishing marginal risk per additional import, with the cap preventing fan-out from dominating the score.

Because the formula differs, fallow's MI scores are **not directly comparable** to Visual Studio or SonarQube MI scores.

</Accordion>

### Fan-in and fan-out

**Fan-in** (importers): Number of files that import this file. High fan-in means high **blast radius** — changes to this file affect many dependents.

**Fan-out** (imports): Number of files this file directly imports. High fan-out indicates high coupling and change propagation risk.

| Metric | Interpretation | Action |
|:-------|:---------------|:-------|
| Fan-in > 20 | Many dependents — this is a critical file | Extra review before changes, consider interface stability |
| Fan-out > 15 | Many imports — high coupling | Consider breaking into smaller modules |

**Research:** Fan-in/fan-out coupling metrics originate from [Henry and Kafura (1981)](https://doi.org/10.1109/TSE.1981.231113), who defined information flow complexity as `length × (fan_in × fan_out)²`. The concept was refined in the [Chidamber & Kemerer OO metrics suite (1994)](https://doi.org/10.1109/32.295895), where Coupling Between Objects (CBO) captures the same idea for class-level dependencies.

### Dead code ratio

Fraction of value exports with zero references (0.0–1.0). Type-only exports (interfaces, type aliases) are excluded from both numerator and denominator to avoid inflating the ratio in well-typed codebases.

| Value | Interpretation | Action |
|:------|:---------------|:-------|
| 0 | All exports are used | No action needed |
| 0.01–0.3 | Some unused exports | Run `fallow check --unused-exports` to identify them |
| 0.3+ | Significant dead code | Run `fallow fix` to auto-remove unused exports |

## Hotspot metrics

Hotspots identify files that are both **complex** and **frequently changing** — the intersection where bugs are most likely to occur and where refactoring yields the highest return. Available via [`fallow health --hotspots`](/cli/health).

The core insight: complex code that never changes is stable (leave it alone); simple code that changes often is fine (easy to modify); but complex code that changes often is where defects concentrate.

**Research:** The churn × complexity methodology was pioneered by Michael Feathers and systematized by Adam Tornhill in [*Your Code as a Crime Scene*](https://pragprog.com/titles/atcrime/your-code-as-a-crime-scene/) (2015) and [*Software Design X-Rays*](https://pragprog.com/titles/atevol/software-design-x-rays/) (2018).

### Hotspot score

**Formula:**

```
normalized_churn = weighted_commits / max_weighted_commits   (0..1)
normalized_complexity = complexity_density / max_density      (0..1)
score = normalized_churn × normalized_complexity × 100       (0..100)
```

Uses within-project max normalization — scores are relative to the most churned and most complex file in your project. The multiplicative relationship means a file needs **both** high churn AND high complexity to score highly; either dimension alone produces a low score.

| Score | Interpretation | Action |
|:------|:---------------|:-------|
| 70–100 | Critical hotspot | Prioritize refactoring in next sprint |
| 40–70 | Moderate hotspot | Schedule review, reduce complexity |
| 0–40 | Low risk | Monitor |

**Example** — `types.ts` has the highest hotspot score because it combines high complexity density (0.20) with frequent changes (47 commits). The `accelerating` trend means it's getting *more* active:

```ansi
● Hotspots (3 files, since 6 months)

   72.1    src/types.ts
           47 commits  +1,892 -847 churn  0.20 density  42 fan-in  accelerating

   38.4    src/helpers/util.ts
           23 commits  +412 -198 churn  0.22 density  35 fan-in  stable

   12.7    src/locales/en.ts
            8 commits  +145 -32 churn  0.35 density   2 fan-in  cooling
```

### Weighted commits

Recency-weighted commit count using exponential decay with a 90-day half-life. A commit yesterday contributes ~1.0; a commit 90 days ago contributes ~0.5; a commit 180 days ago contributes ~0.25.

**Research:** Recency-weighted change metrics for defect prediction were introduced by [Graves, Karr, Marron, and Siy (2000)](https://doi.org/10.1109/32.859533), who demonstrated that recent changes are significantly more predictive of faults than older changes. The empirical link between code churn and defect density was further established by [Nagappan and Ball (2005)](https://doi.org/10.1145/1062455.1062514) at Microsoft.

### Churn trend

Compares commit frequency in the first half vs the second half of the analysis window:

| Trend | Meaning | Action |
|:------|:--------|:-------|
| `accelerating` | Recent commits > 1.5× older half — activity increasing | High priority if complexity is also high |
| `stable` | Balanced commit frequency | Normal — monitor score |
| `cooling` | Recent commits < 0.67× older half — activity decreasing | Lower priority — file is stabilizing |

## Duplication metrics

From [`fallow dupes`](/cli/dupes) output. Fallow uses token-based clone detection with configurable normalization.

**Research:** Clone detection taxonomy and techniques are surveyed in [Roy, Cordy, and Koschke (2009)](https://doi.org/10.1016/j.scico.2009.02.007). Token-based clone detection was pioneered by [Baker (1995)](https://doi.org/10.1109/WCRE.1995.514697). Fallow's approach is token-based with four normalization levels:

| Mode | Clone type | What is normalized |
|:-----|:-----------|:-------------------|
| `strict` | Type-1 | Nothing — exact token match |
| `mild` | Type-2 | Identifiers abstracted |
| `weak` | Type-2+ | Identifiers + literals abstracted |
| `semantic` | Type-2+ | Identifiers + literals + types stripped |

See the [dupes command reference](/cli/dupes) for usage details and examples.

**Example** — two validation modules with identical structure but different field names. Fallow detects these as a clone group in `mild` mode because identifiers are abstracted:

```ansi
● Clone group (42 lines, 2 instances)
  src/validators/userValidator.ts:12-53
  src/validators/orderValidator.ts:8-49

  Family: src/validators/userValidator.ts ↔ src/validators/orderValidator.ts
    3 clone groups, 118 duplicated lines
    → Extract shared validation module
```

### Duplication percentage

Fraction of total source tokens that appear in at least one clone group. Computed over the full analyzed file set before any `--top` truncation.

### Token count / line count

Per-clone-group size metrics. Tokens are language-aware (keywords, identifiers, operators, punctuation). Line count is the span in source lines. Larger clones have higher refactoring value — prioritize groups with the most tokens.

### Clone groups vs clone families

- **Clone group**: A set of 2+ code fragments with identical normalized token sequences at different locations.
- **Clone family**: Multiple clone groups that share the same set of files — indicating systematic duplication patterns (e.g., two modules that are copy-pasted variants of each other). Families suggest **extract-module** refactoring rather than per-function extraction.

## Limitations

Metrics are signals, not verdicts. Be aware of these edge cases:

- **Generated files** (GraphQL codegen, Prisma client, OpenAPI types) may have high complexity density but shouldn't be refactored. Use `health.ignore` patterns to exclude them.
- **Test files** with many `it()`/`test()` blocks can have high cyclomatic complexity from the sheer number of cases. This is usually fine — test suites *should* cover many paths.
- **Configuration files** (Webpack, ESLint configs) may appear as hotspots because they change often during project setup. Their churn typically cools after initial configuration.
- **MI scores are relative** — fallow's formula differs from Visual Studio and SonarQube, so scores are not directly comparable across tools.
- **Hotspot scores are project-relative** — a score of 80 means "worst in your project," not "objectively bad." A well-maintained project may have a top score of 30.
- **Fan-in/fan-out counts modules, not individual imports** — `import { a, b, c } from './utils'` counts as 1 fan-out edge, not 3.

## JSON `_meta` object

When `--explain` is passed (or via MCP), each command's JSON output includes a `_meta` object:

```json
{
  "schema_version": 3,
  "_meta": {
    "docs": "https://docs.fallow.tools/cli/health",
    "metrics": {
      "maintainability_index": {
        "name": "Maintainability Index",
        "description": "Composite score: 100 - (complexity_density × 30) - ...",
        "range": "[0, 100]",
        "interpretation": "higher is better; <40 poor, 40–70 moderate, >70 good"
      }
    }
  }
}
```

This allows AI agents and CI systems to programmatically interpret metric values without consulting external documentation.
