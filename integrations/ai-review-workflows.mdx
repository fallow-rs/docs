When AI tools produce more code, review becomes the bottleneck. Fallow helps by turning the review question from "read everything equally" into "look closely where the evidence says risk is highest."

<Info>
  Fallow does not judge intent. It reports facts: what became unused, what is duplicated, what grew more complex, what crosses a boundary, what landed on a hot path.
</Info>

## Good review workflow

<Steps>
  <Step title="Run an incremental audit">
    ```bash
    fallow audit --base main --format json
    ```

    Returns a verdict (`pass`, `warn`, or `fail`) plus the raw findings. Use it as the default first check on any AI-assisted PR.
  </Step>

  <Step title="Check changed-file dead code">
    ```bash
    fallow dead-code --changed-since main --format json
    ```

    Catches newly orphaned files, unused exports introduced in the patch, and dependencies the patch never actually imports.
  </Step>

  <Step title="Check duplication and health in changed files">
    ```bash
    fallow dupes --changed-since main --format json
    fallow health --changed-since main --format json
    ```

    Surfaces clone groups involving the patch and flags complexity growth in the touched files.
  </Step>

  <Step title="If Runtime is enabled, merge execution evidence">
    ```bash
    fallow health --production-coverage ./coverage --changed-since main --format json
    ```

    Touched hot paths are promoted to a `hot-path-changes-needed` verdict so reviewers escalate carefully.
  </Step>
</Steps>

## What to look for

- new exports with zero importers
- clone groups involving the changed files
- rising complexity in already complex files
- boundary violations introduced by the patch
- hot-path changes that deserve slower review
- cold-path code that may now be safe to remove

## Team policy ideas

<CardGroup cols={2}>
  <Card title="Fail CI on new boundary violations" icon="shield-halved">
    Stop architecture drift from accumulating patch by patch.
  </Card>
  <Card title="Warn on hot-path changes" icon="fire">
    Use runtime evidence to flag risky edits without blocking cleanup work.
  </Card>
  <Card title="Require manual review for complexity growth" icon="chart-line">
    Tight cognitive complexity thresholds catch quietly growing files.
  </Card>
  <Card title="Auto-fix low-risk dead exports" icon="wand-magic-sparkles">
    Let `fallow fix` remove unused exports and dependencies before human review.
  </Card>
</CardGroup>

## Why this works well with AI

AI speeds up patch generation. Fallow speeds up patch judgment.

Models produce syntactically valid code while missing whole-codebase facts: that the helper already exists, that this file is already too complex, that the deletion removed a hot-path function. Fallow supplies those facts deterministically.
